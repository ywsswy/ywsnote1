### 🟢 基础篇（Easy）- 考察日常编码基本功
这部分主要考察候选人是否真正写过Go代码，是否了解Go最基本的数据结构和语法特性。

**1. 数组（Array）和切片（Slice）的区别是什么？切片的扩容机制是怎样的？**
*   **考察点**：Go中最常用的数据结构，值类型与引用类型的区别。
*   **参考要点**：
    *   数组是固定长度的，值类型；切片是动态长度的，是对数组的引用（底层结构包含指针、长度len、容量cap）。
    *   扩容机制（Go 1.18前后有变化）：1.18之前，容量小于1024时翻倍，大于1024时每次增加25%；1.18之后，引入了更平滑的扩容公式，避免了阈值处的突变。

**2. `make` 和 `new` 的区别是什么？**
*   **考察点**：内存分配的基础认知。
*   **参考要点**：
    *   `new` 用于分配内存，返回的是指针，内存被清零（零值）。
    *   `make` 仅用于 `slice`、`map`、`channel` 的初始化，返回的是这三种类型本身（引用），并且会完成内部结构的初始化。

**3. `defer` 的执行顺序是怎样的？如果 `defer` 中修改了返回值，会影响最终结果吗？**
*   **考察点**：控制流和函数生命周期。
*   **参考要点**：
    *   LIFO（后进先出），类似栈。
    *   是否影响返回值取决于返回值是否是**命名返回值**。如果是命名返回值，`defer` 中修改会影响最终结果；如果是匿名返回值，则不会（因为`return`时会先赋值给一个临时变量）。

**4. Go中的 `map` 是并发安全的吗？如果要在并发环境下使用，应该怎么做？**
*   **考察点**：并发编程的防坑意识。
*   **参考要点**：
    *   原生 `map` 不是并发安全的，并发读写会导致 `fatal error: concurrent map read and map write`，程序直接崩溃（无法被recover）。
    *   解决方案：加读写锁（`sync.RWMutex`）或者使用 `sync.Map`。

### 🟡 进阶篇（Medium）- 考察并发编程与核心机制
这部分主要考察候选人对Go语言核心卖点（并发）的掌握程度，区分“会写Go”和“懂Go”的候选人。

**1. Goroutine 和操作系统的线程有什么区别？为什么 Goroutine 更轻量？**
*   **考察点**：并发模型的基础概念。
*   **参考要点**：
    *   Goroutine 是用户态的协程，由 Go Runtime 调度；线程是内核态的，由 OS 调度。
    *   轻量体现在：初始栈空间小（通常2KB，可动态扩缩容），上下文切换成本低（不需要陷入内核态，只保存少量寄存器状态）。

**2. Channel 的无缓冲（unbuffered）和有缓冲（buffered）有什么区别？向一个已关闭的 channel 读写会发生什么？**
*   **考察点**：CSP并发模型和Channel的边界情况。
*   **参考要点**：
    *   无缓冲：同步通信，发送和接收必须同时准备好，否则阻塞。有缓冲：异步通信，缓冲区满前发送不阻塞，空前接收不阻塞。
    *   向已关闭的 channel 写：`panic`。
    *   从已关闭的 channel 读：先读完缓冲区剩余数据，读完后继续读会得到零值（可通过 `val, ok := <-ch` 的 `ok` 来判断是否关闭）。
    *   关闭已关闭的 channel：`panic`。

**3. `context` 包的作用是什么？通常在什么场景下使用？**
*   **考察点**：工程实践中链路控制的规范。
*   **参考要点**：
    *   用于在 Goroutine 之间传递截止日期（Deadline）、取消信号（Cancel）以及请求范围的值（Value）。
    *   常见场景：RPC调用超时控制、HTTP请求上下文传递、级联取消后台任务。

### 🔴 深入篇（Hard）- 考察底层原理与性能调优
对于实习生来说，能回答出这部分说明其有很强的自驱力和技术深度，可以直接给高评价。特别是作为基础架构组，对底层的理解非常重要。

**1. 请详细描述一下 Go 的 GMP 调度模型？**
*   **考察点**：Go Runtime 核心机制。
*   **参考要点**：
    *   **G (Goroutine)**：协程任务。
    *   **M (Machine)**：操作系统线程。
    *   **P (Processor)**：逻辑处理器，包含本地运行队列（Local Queue）。
    *   调度流程：P 绑定 M 执行 G；本地队列满时放入全局队列；M 阻塞（如系统调用）时，P 会与 M 解绑（Hand off）并寻找新的 M；Work Stealing 机制（P 的本地队列空时，从其他 P 偷取一半的 G）。

**2. 什么是内存逃逸（Memory Escape）？什么情况下会发生内存逃逸？**
*   **考察点**：编译原理与性能优化的结合。
*   **参考要点**：
    *   本该分配在栈上的变量，因为生命周期超出了函数范围或大小未知等原因，被编译器分配到了堆上。堆分配会增加 GC 压力。
    *   常见场景：返回局部变量的指针；向 `interface{}` 赋值（如 `fmt.Println`）；闭包引用外部变量；切片/Map的容量在编译期未知或过大。

**3. 简单介绍一下 Go 的垃圾回收（GC）机制？什么是三色标记法？**
*   **考察点**：内存管理。
*   **参考要点**：
    *   Go 使用并发的标记-清除（Mark and Sweep）算法。
    *   三色标记法：白（未访问）、灰（已访问但子节点未访问）、黑（已访问且子节点已访问）。
    *   为了解决并发标记时的漏标问题，引入了**混合写屏障**（Hybrid Write Barrier）机制。

**4. `sync.Map` 是如何实现并发安全的？它适用于什么场景？**
*   **考察点**：高级并发数据结构的底层设计。
*   **参考要点**：
    *   空间换时间，读写分离。底层包含 `read` map（只读，原子操作，无锁）和 `dirty` map（包含所有数据，加互斥锁）。
    *   适用场景：读多写少，或者各个 Goroutine 读写不相交的键。如果是写多读少的场景，性能不如普通的 `map` + `RWMutex`。
